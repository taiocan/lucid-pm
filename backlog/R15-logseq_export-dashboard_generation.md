# R15: `logseq_export` Stage 9 — Schema-Driven Dashboard.md Generation

**Tier**: Refine  
**Depends on**: F2 (logseq_export), F11 (project_schema)  
**Event spine impact**: None  
**Status**: BACKLOG

**Trigger type**: HUMAN_APPROVED_EVOLUTION

---

**The problem**

`lucid export` writes item pages to the Logseq output directory but produces no discovery layer. A PM who opens the exported graph for the first time sees a flat list of pages with no entry point. The five standard queries (Pending Milestones, Active Work Packages, Open Tasks, Risk Pages, Active Stakeholders) that make the graph operationally useful must be hand-authored or copied from the demo repository.

A static `Dashboard.md` exists in the demo repo but is not generated by `lucid export`. New projects have no dashboard unless the PM manually creates one. If the project schema uses custom type names (e.g., `Workstream` instead of `WorkPackage`), a copied static dashboard will silently query nothing.

F11 deferred `schema_views` (generating Logseq queries from a `views:` schema block). This refinement is a limited first step: a fixed set of standard operational queries, with type slugs derived from the loaded schema vocabulary rather than hardcoded strings.

---

**What needs to change**

- `lucid export` reads the loaded schema's `pageTypes` and `blockTypes` and generates `Dashboard.md` in the output directory.
- Type slugs in the generated query blocks are derived from the canonical schema key (lowercased), not hardcoded. If the schema names `WorkPackage` as `Workstream`, the Active Work Packages query uses `"workstream"`.
- The five standard query sections generated:
  1. **Pending Milestones** — pageType whose canonical key matches the schema's milestone-equivalent type (or is absent if no such type exists)
  2. **Active Work Packages** — pageType matching schema's WP-equivalent type
  3. **Open Tasks** — blockType Task; marker set read from `blockTypes.Task.markers`
  4. **Risk Pages** — pageType matching schema's risk-equivalent type
  5. **Active Stakeholders** — pageType matching schema's stakeholder-equivalent type
- A query section is omitted if the corresponding type is not present in the loaded schema.
- **If `Dashboard.md` already exists in the output directory, it is not overwritten** — user customisation is preserved.
- The demo's `logseq/pages/Dashboard.md` is annotated with a provenance comment identifying it as generated from schema; it is not a maintained source of truth.

**What does NOT change**

- Item page generation logic
- Event spine: no new events; `ExportCompleted` payload unchanged
- Schema loading path (already loaded at export startup)

**Open design questions for Stage 2**

1. How are "equivalent types" resolved? Should the code match by canonical key name (case-insensitive: `WorkPackage` → WP query), or should the schema declare a `role:` field (e.g., `role: work-package`) that export reads? Key-matching is simpler but breaks if a project uses a fully custom vocabulary with no recognisable names.
2. If none of the five standard types are found in the schema, should export still write an empty Dashboard.md (with a comment), or skip Dashboard generation entirely?
3. Should the `query-properties::` hints (e.g., `[:page :status :deadline]`) above each query block be hardcoded per query, or also read from the schema renderer config?
